# UI 设计规范

## 目标

献血管理系统面向血站日常操作，界面应当稳定、清晰、克制，优先支持高频录入、查询、审核和统计查看。设计不追求装饰性，重点是让用户快速判断当前状态、完成操作、发现错误。

## 设计原则

1. 信息优先：页面先呈现用户要处理的数据，再呈现解释文字。
2. 操作明确：按钮、输入框、筛选项和状态必须有清楚的视觉区别。
3. 反馈及时：登录、注册、保存、删除、查询失败都必须给出明确提示。
4. 布局稳定：控件宽度、高度和间距保持固定规则，避免输入内容后页面跳动。
5. 可访问：文字和背景保持足够对比度，错误信息不能只依赖颜色表达。

## 视觉风格

整体风格采用医疗业务系统的克制风格：

- 背景：浅米白或浅灰，减少纯白大面积刺眼感。
- 主色：深红，用于主按钮、重点边框、当前选中项。
- 文字：深灰或近黑，用于正文和标题。
- 面板：白色，用于表单、表格、统计块。
- 边框：浅灰，用于输入框、表格和分隔线。

建议颜色：

| 用途 | 色值 |
| --- | --- |
| 主色 | `#AE262D` |
| 正文 | `#23262E` |
| 深色侧栏 | `#23262E` |
| 页面背景 | `#F6F4EF` |
| 面板背景 | `#FFFFFF` |
| 边框 | `#D5D0C6` |
| 成功 | `#1F8A4C` |
| 警告 | `#B7791F` |
| 错误 | `#C53030` |

## 字体与字号

使用系统默认无衬线字体，保证跨平台显示稳定。

| 场景 | 建议 |
| --- | --- |
| 页面标题 | 26-32px，加粗 |
| 区块标题 | 18-22px，加粗 |
| 正文 | 14-16px |
| 表格文字 | 13-15px |
| 按钮文字 | 14-16px |
| 错误提示 | 13-14px |

标题用于区分层级，不用于装饰。表格、表单、弹窗内不要使用过大的标题。

## 布局

主界面采用左侧导航 + 右侧内容区：

- 左侧导航固定宽度，放置业务模块入口。
- 右侧内容区显示当前模块页面。
- 页面内容使用统一边距，建议 32-42px。
- 表单宽度保持稳定，输入框建议宽度 280-360px。
- 列表和表格优先占满可用宽度。

弹窗用于短流程操作，例如注册、确认删除、简单编辑。复杂业务录入应使用独立页面或右侧内容区，不堆在多层弹窗里。

## 表单

表单必须包含：

- 清楚的字段标签。
- 稳定的输入框宽度和高度。
- 必填字段校验。
- 错误提示。
- 提交按钮和取消入口。

注册表单规则：

- 用户名不能为空。
- 密码不能为空。
- 邮箱不能为空，并校验常见邮箱格式。
- 手机号可以为空；填写时必须是 11 位数字。

错误提示应直接说明问题，例如“邮箱格式不正确”，不要只显示“提交失败”。

## 按钮

按钮分为三类：

| 类型 | 用途 | 样式 |
| --- | --- | --- |
| 主按钮 | 登录、注册、保存、创建 | 深红底白字 |
| 次按钮 | 取消、返回、重置 | 背景透明或浅色 |
| 危险按钮 | 删除、停用 | 错误色或深红强调 |

一个区域内只保留一个主按钮。多个按钮并列时，主按钮放在右侧或最容易确认的位置。

## 表格与列表

业务数据、订单记录、日志审计优先使用表格。

表格应包含：

- 清晰表头。
- 固定行高。
- 空状态提示。
- 加载失败提示。
- 分页或限制展示数量。
- 可筛选字段放在表格上方。

状态字段不要只显示数字，应显示中文标签，例如“启用”“停用”“待处理”“已完成”。

## 状态与反馈

系统至少需要这些反馈：

- 成功：保存成功、注册成功、登录成功。
- 警告：账号停用、密码错误、输入格式错误。
- 错误：数据库连接失败、保存失败、查询失败。
- 空状态：暂无数据。

提示文字应短而具体。用户能根据提示知道下一步该做什么。

## 导航

左侧导航建议包含：

1. 业务数据
2. 订单记录
3. 分类管理
4. 评论互动
5. 统计报表
6. 系统日志

当前模块需要有选中态。未实现模块可以保留入口，但点击后必须给出明确提示，不能无反应。

## 可访问性

- 文本和背景要有明显对比。
- 按钮必须能通过键盘触发。
- 登录和注册按钮支持回车提交。
- 输入框不能过窄，至少能完整显示常见用户名和邮箱。
- 错误提示不能只靠红色表达，必须有文字说明。

## 文案

文案使用简洁中文：

- “登录”
- “注册账号”
- “请输入用户名和密码”
- “用户名或密码错误”
- “账号已停用”
- “邮箱格式不正确”
- “手机号格式不正确”
- “数据库连接失败，请检查服务和配置。”

避免使用含糊文案，例如“失败”“错误”“异常”。

## 实现约束

- 优先使用 Swing 原生组件。
- 不引入新的 UI 依赖。
- 不为单个页面创建复杂主题框架。
- 可复用按钮、输入框、提示样式，但不要提前抽象完整组件库。
- 每次新增页面后至少运行一次编译或测试。

## 检查清单

提交 UI 改动前检查：

- 输入框宽度足够。
- 按钮可见且可点击。
- 错误提示清楚。
- 页面没有控件重叠。
- 字号和间距与现有页面一致。
- 登录、注册或保存等关键路径已验证。
