# 献血管理系统技术设计文档

## 1. 设计目标

献血管理系统面向课程实训和答辩演示，提供用户、档案、血液库存、
用血申请、评论、日志和统计报表的统一管理界面。

技术设计需要满足以下目标：

- 使用 Java Swing 提供可直接运行的桌面界面。
- 使用 JDBC 和 DAO 封装 MySQL 数据访问。
- 使用 MongoDB 保存日志、评论和扩展详情。
- 将库存、申请和用户关系保留在具有事务能力的 MySQL 中。
- 使用预编译语句、密码哈希、角色校验和日志记录建立安全基线。
- 使用自动化测试保护核心业务规则和回归场景。

## 2. 总体架构

系统采用桌面界面、业务服务、数据访问和数据库四层结构。

```text
Java Swing UI
  ├── LoginFrame
  └── DashboardFrame
          │
          ▼
Service
  ├── AuthService
  ├── BusinessService
  └── RecommendService
          │
          ▼
DAO
  ├── MySQL DAO ── BaseDAO ── HikariCP ── MySQL
  └── MongoDB DAO ── MongoBaseDAO ── MongoClient ── MongoDB
```

界面层不直接执行 SQL 或 MongoDB 查询。业务层负责输入校验、权限、状态
和跨数据源协调，DAO 负责具体读写。

## 3. 技术选型

| 技术 | 用途 | 选择原因 |
| --- | --- | --- |
| Java 17 | 应用开发 | 支持 record、文本块和成熟 JDBC 生态 |
| Java Swing | 桌面界面 | JDK 原生桌面 UI，无需引入额外前端框架 |
| Maven | 构建和依赖 | 统一编译、测试和依赖版本 |
| JDBC | MySQL 访问 | 满足课程要求，显式展示 SQL 和事务 |
| HikariCP | MySQL 连接池 | 复用连接，简化连接生命周期管理 |
| MySQL 8.0 | 核心业务数据 | 提供外键、事务、行锁、视图、存储过程和触发器 |
| MongoDB 5.0 | 扩展数据 | 适合日志、嵌套属性、标签数组和聚合统计 |
| BCrypt | 密码哈希 | 为密码提供随机盐和可调整成本 |
| JUnit 5 | 自动化测试 | 支持服务层、结构和压力测试 |

## 4. 项目模块

### 4.1 程序入口

`Main` 负责记录启动日志，并在 Swing 事件分发线程中打开登录窗口。

### 4.2 认证模块

`AuthService` 提供注册和登录功能。

主要职责：

- 清理用户名、邮箱和手机号输入。
- 验证必填项、邮箱和手机号格式。
- 检查重复用户名。
- 使用 BCrypt 生成和校验密码哈希。
- 检查账号启用状态。
- 写入登录成功和失败日志。
- 登录成功后生成只包含必要用户信息的会话对象。

### 4.3 业务模块

`BusinessService` 作为界面和 DAO 之间的主要边界，统一表达业务规则。

功能范围：

- 用户联系方式、实名档案、角色和状态维护。
- 分类创建、编辑、树形关系校验和删除保护。
- 库存批次创建、编辑、删除和扩展详情协调。
- 评论、评分、标签和删除权限。
- 用血申请创建、查询、编辑、审批、取消和删除。
- 热度、评分、月度报表、用户报表和日志审计。
- 业务操作日志记录。

业务操作返回统一结果，包含成功状态、用户可读消息和业务 ID。

### 4.4 推荐模块

`RecommendService` 将 MySQL 库存、分类和订单数据与 MongoDB 详情、
评分和行为统计合并为库存洞察。

推荐顺序：

1. 只保留状态可用的库存批次。
2. 优先用户曾经申请或访问过的分类。
3. 在同一优先级中按行为热度倒序。
4. 继续按平均评分倒序。
5. 返回“同类库存”、“近期关注度较高”或“可用库存”理由。

## 5. 数据访问设计

### 5.1 MySQL 基础访问

`BaseDAO` 提供以下通用能力：

- 获取 HikariCP 数据库连接。
- 执行插入并返回自增主键。
- 执行更新和删除。
- 查询单行或列表数据。
- 将 JDBC `ResultSet` 转换为保留字段顺序的键值映射。

所有用户输入通过 `PreparedStatement` 参数绑定，不拼接进 SQL。

### 5.2 MySQL DAO

| DAO | 主要职责 |
| --- | --- |
| `UserDAO` | 用户创建、登录查询、联系方式、角色和状态 |
| `ProfileDAO` | 实名档案创建、查询、编辑和删除 |
| `CategoryDAO` | 分类层级查询和维护 |
| `ItemDAO` | 库存批次创建、查询、编辑和状态维护 |
| `OrderDAO` | 用血申请、月度报表、审批事务和库存恢复 |

### 5.3 MongoDB DAO

| DAO | 集合 | 主要职责 |
| --- | --- | --- |
| `DetailDAO` | `item_details` | 库存详情更新或插入、查询和删除 |
| `CommentDAO` | `comments` | 评论、评分、标签、查询和评分聚合 |
| `LogDAO` | `action_logs` | 用户行为写入、最近行为和热度聚合 |
| `SystemLogDAO` | `system_logs` | 登录和系统日志写入、查询和审计聚合 |

## 6. 数据库设计

### 6.1 MySQL 设计边界

MySQL 保存用户、分类、库存批次、用血申请和实名档案。

关系约束：

- 用户名和邮箱唯一。
- 档案与用户为一对一关系。
- 分类使用 `parent_id` 表示父子结构。
- 库存批次必须关联有效分类。
- 用血申请必须关联有效用户和库存批次。
- 库存数量不得为负数，申请数量必须大于 0。
- 用户、库存和申请状态只能使用定义范围内的值。

### 6.2 MongoDB 设计边界

MongoDB 保存扩展属性、数组和日志。

- `item_details` 包含描述、图片列表和 `metadata` 嵌套文档。
- `comments` 包含评论、评分和标签数组。
- `action_logs` 包含行为类型、停留时间和客户端信息。
- `system_logs` 包含日志类型、级别、消息和操作详情。

MySQL 的业务 ID 在 MongoDB 中作为字符串引用字段。

### 6.3 查询与索引

MySQL 为用户名、邮箱、父分类、库存分类、状态、申请用户、
申请批次和时间建立索引。

MongoDB 使用“过滤字段 + 时间倒序”组合索引支持最近日志和评论查询。

## 7. 核心业务流程

### 7.1 注册流程

```text
输入用户名、密码、邮箱和手机号
  → 必填项和格式校验
  → 查询用户名是否存在
  → BCrypt 生成密码哈希
  → MySQL 创建 USER 角色账号
  → 返回注册结果
```

### 7.2 登录流程

```text
输入用户名和密码
  → 按用户名查询 MySQL
  → 检查账号状态
  → BCrypt 校验密码
  → MongoDB 写入登录日志
  → 生成会话并进入主界面
```

失败时不向界面暴露“用户不存在”或“密码错误”的区别，
统一使用“用户名或密码错误”。

### 7.3 库存创建流程

```text
管理员输入库存基本信息和详情
  → 检查名称、分类和数量
  → MySQL 创建库存主记录
  → 使用主记录 ID 更新或插入 MongoDB 详情
  → 刷新库存列表
```

MySQL 记录是库存的权威主数据，MongoDB 详情不参与库存数量事务。

### 7.4 用血申请流程

```text
用户选择库存批次并输入数量
  → 检查库存存在
  → 检查库存状态可用
  → 检查申请数量不超过库存
  → MySQL 创建待审批申请
  → MongoDB 写入用户行为日志
```

### 7.5 审批流程

```text
管理员选择待审批申请
  → 开启 MySQL 事务
  → SELECT ... FOR UPDATE 锁定申请和库存
  → 重新检查申请、库存状态和数量
  → 更新申请为已完成
  → 扣减库存数量
  → 提交事务
```

任何一步失败时回滚。库存扣减到 0 时，库存状态更新为不可用。

## 8. 事务设计

### 8.1 事务边界

事务仅包含必须同时成功或失败的 MySQL 操作。

- 审批申请：更新申请状态和扣减库存。
- 删除已完成申请：恢复库存和删除申请。
- 创建申请：显式提交插入。

MongoDB 日志和详情不加入 MySQL 事务，系统不引入分布式事务。

### 8.2 并发保护

审批时使用行级锁防止并发操作同时基于旧库存数量通过校验。

事务内重新读取并检查：

- 申请是否仍为待审批。
- 库存是否仍为可用。
- 库存数量是否仍足够。

## 9. 权限设计

### 9.1 角色边界

| 功能 | 管理员 | 普通用户 |
| --- | --- | --- |
| 查看库存 | 允许 | 允许 |
| 维护库存 | 允许 | 不允许 |
| 查看推荐 | 允许 | 允许 |
| 创建用血申请 | 允许 | 允许 |
| 审批用血申请 | 允许 | 不允许 |
| 维护分类 | 允许 | 不允许 |
| 维护用户 | 允许 | 只能维护自己的档案 |
| 查看系统日志 | 允许 | 不允许 |
| 发表评论 | 允许 | 允许 |

### 9.2 对象归属权

- 普通用户只能编辑或删除自己的待审批申请。
- 普通用户只能删除自己的评论。
- 普通用户只能维护自己的联系方式和档案。
- 普通管理员不能修改用户角色。
- 只有超级管理员可修改其他用户的角色。
- 不允许删除当前登录用户或超级管理员。

## 10. 界面设计

### 10.1 登录界面

登录窗口包含品牌区域、用户名、密码、登录按钮和注册入口。

登录按钮是窗口默认按钮，用户可以按回车提交。数据库异常显示
统一且可操作的错误提示。

### 10.2 主界面

主界面由左侧导航和右侧内容区域组成。导航按用户角色生成。

管理员功能：

- 业务数据。
- 推荐批次。
- 订单记录。
- 用户管理。
- 分类管理。
- 统计报表。
- 系统日志。

普通用户功能：

- 业务数据。
- 推荐批次。
- 订单记录。
- 我的档案。
- 统计报表。

### 10.3 表格和详情

- 列表使用表格显示。
- 内部 ID 列用于定位数据，对用户隐藏。
- 状态显示中文文案，不显示数字编码。
- 状态、分类、血型、角色和日志级别提供筛选。
- 双击表格行打开详情页。
- 删除和登出等操作需要确认。

## 11. 统计和跨数据库设计

### 11.1 MySQL 报表

月度报表通过 `sp_monthly_report` 按年份和月份查询，并按分类统计：

- 申请数量。
- 已完成用量。
- 待审批数量。
- 已完成数量。
- 已取消数量。

### 11.2 MongoDB 报表

- `action_logs` 按库存批次分组，统计操作热度。
- `comments` 按批次分组，统计评论数和平均评分。
- `system_logs` 按类型和级别分组，统计数量和最近时间。

### 11.3 跨数据库合并

库存洞察通过 MySQL `item_id` 将以下数据合并：

- MySQL 库存名称、分类、数量和状态。
- MySQL 申请数量。
- MongoDB 详细描述和血型。
- MongoDB 评论数和平均评分。
- MongoDB 行为次数。

合并在服务层完成，不要求数据库之间建立直接连接。

## 12. 异常和日志设计

### 12.1 数据库异常

JDBC 配置、连接、查询和更新异常转换为项目统一的 `DBException`。

界面不直接向用户显示驱动堆栈、SQL 或密码等底层信息，而是显示
“数据库连接失败”或“保存失败”等操作性提示。

### 12.2 应用日志

SLF4J 和 Logback 记录程序启动和 Java 运行日志。

### 12.3 业务日志

MongoDB 记录：

- 登录成功、登录失败和停用账号尝试。
- 创建、修改和删除用血申请。
- 创建和删除评论。
- 申请状态变更和其他关键操作。

## 13. 安全设计

### 13.1 密码

- 数据库只保存 BCrypt 哈希。
- 每次生成哈希时使用随机盐。
- 登录时使用 BCrypt 校验，不解密密码。
- 登录日志不记录明文密码。

### 13.2 SQL 注入防护

DAO 使用 `PreparedStatement` 或 `CallableStatement` 绑定用户输入和查询参数。

### 13.3 数据安全

- 演示邮箱使用 `.test` 域名。
- 证件号和地址是明确的测试值。
- 清理脚本与初始化脚本分开，执行前可明确识别风险。

## 14. 性能设计

- HikariCP 复用 MySQL 连接。
- 高频等值查询、关联字段和时间范围建立索引。
- MongoDB 最近日志查询使用过滤字段和时间的组合索引。
- 日志和评论列表使用读取数量上限。
- 报表在数据库端分组和聚合。
- 库存审批事务只包含必要的查询和更新，减少持锁时间。

性能数据、执行计划和压力测试结果见《性能优化报告》。

## 15. 测试设计

### 15.1 服务层测试

使用可控 DAO 替身验证认证、分类、库存、评论、用户、申请、
统计和推荐的业务规则。

### 15.2 DAO 结构测试

检查课程要求的 DAO 类、基类继承关系和显式事务方法。

### 15.3 UI 回归测试

验证删除数据后统计行处理和 MySQL 日期时间值转换。

### 15.4 压力测试

使用 50 个并发工作线程向 MongoDB 写入 10000 条系统日志，按唯一
`run_id` 验证数量并清理本次测试数据。

## 16. 部署和配置

### 16.1 运行组件

- Java 17 运行环境。
- MySQL 8.0 服务。
- MongoDB 5.0 服务。

### 16.2 配置文件

`db.properties` 包含 MySQL URL、账号、密码、驱动和 MongoDB URI、数据库名。

当前配置用于本地课程 Docker 环境。如部署到共享环境，数据库凭据
应通过环境变量或密钥管理注入。

### 16.3 数据库初始化

MySQL 脚本顺序：

1. 清理数据库。
2. 创建表和索引。
3. 创建视图。
4. 创建存储过程。
5. 创建触发器。
6. 导入演示数据。

MongoDB 脚本顺序：

1. 清理集合。
2. 创建集合和索引。
3. 导入演示文档。

## 17. 设计边界

### 17.1 跨数据库一致性

系统不使用分布式事务。MySQL 保证库存和申请的核心状态，MongoDB
详情和日志属于可补写的扩展数据。

如长期运行环境中 MongoDB 写入失败频繁，可增加对账、重试或补写队列。

### 17.2 应用形式

系统为桌面应用，不包含 Web 服务、移动客户端、第三方医院接口、
短信、邮件或真实身份核验。

### 17.3 数据规模

当前设计面向课程演示和小规模使用。如单表或日志集合达到大数据量，
需要根据慢查询、内存和存储指标调整分页、索引、归档和报表读模型。

## 18. 设计结论

系统以 MySQL 承担核心关系和事务，以 MongoDB 承担详情、评论和日志，
通过服务层统一表达权限、状态、数量和跨数据源协调规则。

设计覆盖课程要求的 JDBC、DAO、连接池、事务、视图、存储过程、触发器、
MongoDB 聚合、跨数据库查询、密码哈希、日志审计和自动化测试。
